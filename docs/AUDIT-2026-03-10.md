## 1. CRITICAL — Would Fail a Security Audit

### CRIT-1: Unauthenticated subscription endpoints write to `users` and `subscribers` tables

**Files**: `routes/subscriptions.js:148`, `routes/subscriptions.js:80`, `routes/subscriptions.js:553`
**Impact**: HIGH — attacker can create user accounts, enumerate subscriber emails, brute-force promo codes

Three subscription endpoints have **zero authentication**:
- `POST /api/subscriptions/create` (line 148) — **creates user accounts**, inserts into `subscribers`, `users`, `password_reset_tokens`, runs promo codes. No auth, no rate limit, no CAPTCHA.
- `POST /api/subscriptions/promo/validate` (line 80) — brute-forceable promo code validation. No rate limit. Responds with discount details on success.
- `GET /api/subscriptions/status` (line 553) — leaks subscription status by email. Pass any email, get back subscription tier, status, and dates.

The `/create` endpoint is particularly dangerous: it creates a real user in the `users` table with a password hash and links it to a subscriber record. An attacker could:
1. Flood the users table with garbage accounts
2. Enumerate valid promo codes via `/promo/validate`
3. Check if any email is a subscriber via `/status`

**Remediation**: Add rate limiting (strict — 3/15min) to `/create` and `/promo/validate`. Add rate limiting to `/status`. Consider requiring CAPTCHA or email verification on `/create`.

### CRIT-2: Subscription routes have no `merchant_id` scoping — global database writes

**File**: `routes/subscriptions.js:422-506`
**Impact**: HIGH — multi-tenant isolation bypass

The `POST /subscriptions/create` endpoint writes to `subscribers`, `users`, `promo_codes`, and `password_reset_tokens` without any `merchant_id` filter. The `subscribers` table queries at lines 84, 188, 422, 452, 457, 462, 478, 487, 499, 504, 848, 850 all lack merchant scoping. While subscription management is arguably platform-level (not per-merchant), the promo code validation at line 84 queries `promo_codes` without merchant scoping — a merchant's promo code is usable by any subscriber.

### CRIT-3: 288 innerHTML assignments in frontend JS — systematic XSS surface

**Files**: All files in `public/js/` — 288 `innerHTML` assignments total
**Impact**: MEDIUM-HIGH — stored XSS if Square catalog data contains malicious strings

While many files do use `escapeHtml()`, the grep shows large swaths of unescaped innerHTML in:
- `public/js/logs.js` — 8 innerHTML calls, none escaped (SEC-12, still open)
- `public/js/delivery-settings.js` — innerHTML without escaping (SEC-13, still open)
- `public/js/catalog-audit.js` — 10+ innerHTML calls rendering catalog data (item names, SKUs, categories from Square API)
- `public/js/cycle-count.js` — renders inventory data via innerHTML
- `public/js/deleted-items.js` — renders item data

The data source is Square's catalog API. If a malicious merchant (or compromised Square account) injects `<img onerror=...>` into an item name, it executes in every admin's browser. In a multi-tenant SaaS, tenant A's data should never execute code in tenant B's session.

**Remediation**: Audit every `innerHTML` assignment. Any that interpolate API-sourced data must use `escapeHtml()` or `textContent`.

### CRIT-4: DB-1 remains open — 14+ tables have nullable `merchant_id`

**File**: `database/schema.sql` — multiple tables
**Impact**: MEDIUM — data isolation relies on application code, not database constraints

The schema grep shows tables where `merchant_id` is `INTEGER REFERENCES merchants(id)` without `NOT NULL`:
- `sync_history` (line 148)
- `vendors` (implicit from index)
- `variation_vendors` (implicit from unique constraint)
- `inventory_counts` (implicit from unique constraint)
- `committed_inventory` (implicit from indexes)
- `sales_velocity` (implicit from unique constraint)
- `variation_discount_status` (implicit from unique constraint)

Without `NOT NULL`, a bug in any INSERT statement can create orphaned rows visible to all merchants or invisible to queries filtering by `merchant_id`. This is documented as DB-1 in PRIORITIES.md but remains unfixed. Migration 065 only fixed some tables.

### CRIT-5: Square API version fragmentation — 19 files hardcode `'2025-01-16'`

**Files**: 19 files in `services/loyalty-admin/` hardcode `'2025-01-16'`
**Config**: `config/constants.js:77` defines `API_VERSION: '2025-10-16'`
**Impact**: MEDIUM — Square deprecates old API versions; different endpoints talk to different API versions

The `shared-utils.js` correctly imports from `config/constants.js`, but 19 loyalty-admin service files bypass it entirely and hardcode `'2025-01-16'` in HTTP headers. When Square deprecates `2025-01-16`, these 19 files break while the rest of the app works. The version gap (`2025-01-16` vs `2025-10-16`) is 9 months — response shapes may already differ.

## 2. ARCHITECTURE — Design Decisions That Will Hurt at Scale

### ARCH-1: No distributed locking on cron jobs

**File**: `jobs/cron-scheduler.js`
**Impact**: If you ever run 2+ instances (PM2 cluster, multiple servers), every cron job runs N times simultaneously

15 cron jobs are registered in `cron-scheduler.js`. Zero use `pg_try_advisory_lock` or any distributed locking. The backup job would run twice. The expiry discount job would double-apply discounts. The velocity sync would double-count. The loyalty catchup would double-process orders. This is a **hard blocker** for horizontal scaling.

Documented in ROADMAP.md #5 but still no implementation. For a "multi-tenant SaaS" this is table stakes.

### ARCH-3: `server.js` at 1,188 lines — still a monolith entry point

Despite documentation claiming it was "reduced 66%", server.js is 1,188 lines and growing. Route registration, middleware setup, error handling, email notification, static file serving — all in one file.

### ARCH-4: Two Square API client implementations

- `services/loyalty-admin/square-api-client.js` — HTTP client with 429 retry
- `services/square/square-client.js` — HTTP client with retry/rate-limit
- `middleware/merchant.js` — SDK client via `getSquareClientForMerchant()`

Three different ways to talk to Square. BACKLOG-19 claimed to unify these but two distinct HTTP clients remain alongside the SDK client. Different retry logic, different error handling, different version headers.

### ARCH-5: No health check endpoint

There is no `/health` or `/readiness` endpoint. The `catalog-health` routes are for catalog data, not application health. If this runs behind a load balancer or in a container orchestrator, there's no way to determine if the service is alive, can reach the database, or can reach Square's API.

### ARCH-6: 67 schema tables, 70 migrations — no migration testing

70 migrations and zero migration tests. No rollback scripts. No idempotency checks. A failed migration leaves the database in an unknown state. `schema.sql` and migrations can drift silently — there's no CI step that verifies they produce the same result.

## 4. OPERATIONAL — What Would Wake You Up at 3am

### OPS-1: BACKLOG-61 is a P0 — GMC API broken for 10+ days

`services/gmc/merchant-service.js` uses Google Merchant API v1beta (discontinued Feb 28, 2026). All product upserts are returning 409 ABORTED. The live online store has had **zero organic Google Shopping visibility for 10+ days**. This is documented but not fixed. It's actively costing the business money every day.

### OPS-2: No application health check or readiness probe

No `/health` endpoint. No database connectivity check exposed to monitoring. No Square API reachability check. If the DB connection pool exhausts or Square goes down, the only signal is customer complaints or manually tailing logs.

### OPS-3: No alerting on cron job failures

Cron jobs log errors but there is no external alerting. If the expiry discount job fails silently at 5 AM, discounted items revert to full price and customers are overcharged until someone manually checks logs. Same for loyalty catchup, velocity sync, committed inventory reconciliation.

### OPS-4: `merchants.subscription_status` never auto-transitions

Documented in ROADMAP.md #8 — the status column stays stale at `'trial'` forever. The middleware handles this dynamically, but if anyone queries the column directly (admin dashboards, reports), they get wrong data.

### OPS-5: 18 failing test suites — CI is red

The GitHub Actions workflow exists but the test suite fails. This means either: (a) CI is broken and merges happen despite failures, or (b) CI is ignored. Either way, no automated quality gate.

### OPS-6: PM2 restart recovery gap (BACKLOG-9)

In-memory caches (TTLCache for webhook dedup, velocity dedup, invoice scope cache) are lost on PM2 restart. The webhook dedup cache loss means the first 60 seconds after restart can double-process webhooks. Documented as "no immediate action needed" but it's a real race condition window.

## 5. CODE QUALITY — What Would Embarrass You in a PR Review

### CQ-1: Square API version hardcoded 19 times instead of using constant

`config/constants.js` defines `SQUARE.API_VERSION` as `'2025-10-16'`. Nineteen loyalty-admin files ignore it and hardcode `'2025-01-16'`. This is not a style issue — it's a **9-month version gap** in production API calls.

### CQ-2: `process.env` accessed directly in 20+ service/route files

`config/constants.js` exists but is bypassed. `services/catalog/reorder-service.js` reads 6 env vars directly. `services/delivery/delivery-service.js` reads 2. Webhook handlers read feature flags directly. This defeats centralized configuration.

### CQ-3: Dead code — velocity ternary no-op

`services/square/square-velocity.js:131-132`:
```javascript
const locationId = returnItem.source_line_item_uid ? order.location_id : order.location_id;
```
Both branches return the same value. Dead ternary left in production.

### CQ-4: Dead code — `updateDiscountAppliesTo` is a no-op function

`services/expiry/discount-service.js:660-738` — function accepts `variationIds` but never uses them. The Square API call sends the existing data unchanged. 78 lines of dead code.

### CQ-5: TODO comments left in production code

- `services/webhook-handlers/order-handler/order-loyalty.js:445` — `// TODO: Suggested extraction`
- `services/loyalty-admin/reward-progress-service.js:25` — `// TODO: Refactoring suggestions`
- `services/loyalty-admin/purchase-service.js:353,361` — Two TODOs for extraction

### CQ-6: `hashResetToken` duplicated in `auth.js` and `subscriptions.js`

Identical 3-line SHA-256 hashing function copy-pasted across two route files. Documented in TECHNICAL_DEBT.md but still present.

### CQ-7: Inconsistent response formats

`routes/seniors.js` returns `{ configured, message }`, `{ config }`, `{ members, pagination }` — different shapes per endpoint, none using `success: true/false` wrapper. `routes/subscriptions.js` mixes `{ success: true, plans }` with `{ applicationId, locationId }`. BACKLOG-3 has been open since the beginning.

### CQ-8: `totalSynced` counter inflates on error path

`services/square/square-catalog-sync.js` increments the sync counter before confirming DB write succeeded. Sync summary logs lie about how many items were actually persisted.

### CQ-9: Vendor constraint errors log at ERROR level

`services/square/square-vendors.js` — unique constraint violations (expected race condition) log as ERROR, generating noise in error logs and potential false alerts.

### CQ-10: `filterValidVariations` returns all variations on API failure

`services/expiry/discount-service.js:967-974` — when Square batch-retrieve fails, the catch block includes ALL variations as "valid" to "avoid data loss." This means rate limiting or network errors cause deleted catalog items to receive discount pricing rules.

## 7. HONEST GRADE

### Grade: C+

**Justification**:

**What's genuinely good:**
- Security fundamentals are mostly in place: parameterized queries, HMAC webhook verification, rate limiting on sensitive endpoints, AES-256-GCM token encryption, hashed password reset tokens, CSP headers
- The multi-tenant pattern is enforced in >90% of queries
- The CLAUDE.md documentation is unusually thorough and honest about known issues
- The backlog tracking is meticulous — issues are documented, not hidden
- The webhook architecture is well-designed (idempotency, signature verification, event routing)
- 1,500 tests exist (even if 18 suites are broken)

**What drops the grade:**
- **18 failing test suites (16% of all suites)** — the safety net has holes. CI is effectively theater.
- **Unauthenticated endpoints that write to user tables** — this is a security audit failure for a SaaS product accepting payments.
- **~6,000+ lines of completely untested Square integration code** — this is the core of the product with zero automated verification.
- **19 files hardcoding a 9-month-old API version** while the config says something else — this is how production outages happen.
- **Zero distributed locking on 15 cron jobs** — this makes "multi-tenant SaaS" a single-instance product wearing SaaS clothing.
- **288 innerHTML calls** — even with partial escaping, the surface area is enormous for a multi-tenant app where tenant data is rendered in other tenants' browsers.
- **P0 GMC outage open for 10+ days** with documented impact to the live store.
- **27+ files over 500 lines** with the style guide saying 300 max — the rules exist but aren't followed.

**The documentation gets an A+. The code gets a C-. The average is C+.**

The brutal truth: this codebase was built fast by someone competent and it works for a single store. But the gap between the documented standards and the actual state of the code is wide. The backlog is honest about known issues, which is rare and valuable — but "documented and unfixed" is still unfixed. For a production multi-tenant SaaS handling real money, the test coverage and authentication gaps are the most concerning.
